# [11. Container With Most Water](https://leetcode.com/problems/container-with-most-water/)

## Approach: two pointers from both ends, move inside

Provable that there are no better options when we move one pointer by the area formula itself -- essence of greedy algorithms.

```python
class Solution:
    def maxArea(self, height: List[int]) -> int:
        """
        Summary:
            There are `n` vertical lines drawn s.t. 
            the two endpoints of the `ith` line are
            (i, 0) and (i, height[i])

            Find two lines that together with the x
            axis from a container, s.t. the container
            has the most water.
            

        Args:
            height: arr of length `n`

        Returns:
            max amount of water a container can store.

        formula:
            s = (q-p) * min(height[q], height[p])
            need to find max(s)
        """

        # Greedy?
        p = 0 # left one ptr
        q = len(height) - 1 # right one ptr
        max_s = 0
        while p < q:
            s = (q - p) * min(height[p], height[q])
            max_s = max(max_s, s)
            # how and when to advance q
            if height[p] < height[q]:
                p += 1
            else:
                q -= 1

        return max_s

```



The pattern to recognize has a few ingredients:

**You're optimizing over all pairs (i, j) in an array.** Brute force is O(n²) — try every pair. When you see that, ask: can I narrow the search space by eliminating pairs I know can't win?

**The two-pointer-from-outside-in works when moving one pointer inward can only make things worse for that pointer.** That's what lets you safely discard it. In this problem, the shorter side is the bottleneck — no future pairing can compensate because width only shrinks. So you always discard the bottleneck side.

**The mental framework when you see a new problem like this:**

First, write down the formula you're optimizing. You did this well — `s = (q-p) * min(height[p], height[q])`. Then ask: if I fix one element and move the other pointer inward, which factor can only get worse? If one side is provably stuck (like the min-height side here), that's the one you move.

**Other problems with the same structure:** "two sum" on a sorted array, "3sum", and "trapping rain water" all use variants of this. The common thread is: sorted or structured input, optimizing or searching over pairs, and a provable reason one pointer's remaining options are exhausted.



So next time you see "optimize over pairs + O(n²) is too slow," your first question should be: *is there a pointer I can prove has no better options left?* If yes, two pointers from outside in.
